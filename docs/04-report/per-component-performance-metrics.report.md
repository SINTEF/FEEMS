## Outcome

Four operating-average scalars are now reported on every row of `FEEMSResult.detail_result`
and the matching `ResultPerComponent` proto:

| Field                                  | DataFrame column                       | Proto ID |
|----------------------------------------|----------------------------------------|----------|
| `operating_avg_power_kw`               | `operating avg power [kW]`             | 16       |
| `operating_avg_reversible_power_kw`    | `operating avg reversible power [kW]`  | 17       |
| `operating_avg_efficiency`             | `operating avg efficiency`             | 18       |
| `operating_avg_sfc_g_per_kwh`          | `operating avg SFC [g/kWh]`            | 19       |

All averages are computed over the component's on-state timesteps only â€” never diluted by
shut-down periods. Non-applicable metrics report `0.0` so the DataFrame stays homogeneous
and the proto wire format needs no special-casing.

PTI/PTO is the only bidirectional component: the default field reports the PTO-direction
mean, and the new `operating_avg_reversible_power_kw` field reports the PTI-direction
mean. Both are positive magnitudes.

## Backwards Compatibility

- **Proto:** Fields 16-19 are additive; older clients ignore them, newer clients read `0.0` for messages produced by old code. No tag renumbered.
- **DataFrame:** 4 columns appended at the end. Label-based lookups (`row["running hours [h]"]` etc.) are unaffected. The only positional access in the codebase is via `to_list_for_*_component()`, which is unchanged.
- **`FEEMSResult` dataclass:** New fields default to `0.0`; constructor calls without these kwargs continue to work.
- **`__merge`:** New explicit branch using `max(self, other)` for the 4 fields. Equivalent to the prior `+` behaviour when only one side is non-zero (the typical case during per-component aggregation), and avoids unbounded growth from accumulated sums.

## Lessons / Notes Worth Carrying Forward

1. **Single source of truth for column labels.** Defining `OPERATING_AVG_COLUMNS` as a tuple in `types_for_feems.py` and importing it in both `node.py` and `system_model.py` prevented the column-order drift risk identified in the design doc and made the boiler row "free" to wire up.
2. **Dispatch function placement.** Putting `_assign_operating_avg_metrics` as the *last* step of `get_fuel_emission_energy_balance_for_component` means every branch can populate the 4 fields without changing the existing fuel/emission logic. This isolates the new behaviour and made code review straightforward.
3. **PTI/PTO direction conventions.** Reusing the existing `power_input > 0` (PTI) and `power_input < 0` (PTO) signs from `node.py:284-331` rather than re-defining them prevented sign-flip bugs and matched user expectations stated in the planning conversation.
4. **`0.0` over `None` for N/A metrics.** Keeps the DataFrame dtype-stable (no `object` columns), keeps the proto wire format trivial, and avoids `None`-propagation throughout downstream consumers. Documented explicitly in the design doc and API reference so future contributors don't second-guess.
